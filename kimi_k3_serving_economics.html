Serving Kimi K3 at 1M tokens — the VRAM-per-user math and what it costs

How many users a machine can actively serve at a full 1M-token context is a pure VRAM-capacity question: a fixed model cost, plus a per-user KV-cache cost. Here is the arithmetic, the machines that can do it, and the cost per active user.

Companion to kimi_k3_1m_full_quality.html (the single-instance footprint) and strata_v_design.html (the custom-silicon alternative). Model facts from Moonshot’s K3 announcement; hardware facts from NVIDIA’s current datasheets; Israel prices at FX 3.05 ₪/USD, VAT 18%. Point-in-time, DRAM-crunch pricing — not quotes. Compiled 2026-07.

“How many users can I serve at once?” is not a compute question first — it is a capacity question. Every actively-decoding user must have their entire KV cache resident in VRAM alongside the one shared copy of the model. So the count is just: (VRAM − model) ÷ KV-per-user. Everything else is a footnote.

The point — state it first

Inference is a shared-silicon economy. The model is a fixed cost paid once; each user is a variable cost. So cost-per-user falls as you pack more users onto one memory pool.

Kimi K3 is a fixed ~1.45 TiB resident once, no matter how many users. Each active 1M-token user then adds only ~28 GiB of KV cache. Because the big cost is shared, the same machine serves its 40th user far cheaper than its 1st — and a bigger shared pool is cheaper still per user: ~₪80k/user on a 16-GPU box → ~₪39k on NVL36 → ~₪28k on NVL72. That is the entire economic case for large shared inference silicon — and it is exactly the case that cheap-capacity DDR5 (the STRATA-V converter) pushes to its logical end.

01The per-user arithmetic

Two numbers decide everything, both established from Moonshot’s announced K3 (2.8 T params, native MXFP4, 96 layers, 3:1 KDA:MLA hybrid, 1M context):

CostSizePaid…
Fixed — the model: MXFP4 weights (1,431) + MTP module (18) + vision (1)~1,450 GiBonce, shared by all users
Variable — per active user: KV cache @ 1M tokens, un-quantized FP16~28 GiBper simultaneously-decoding user
Why the per-user KV is only 28 GiB — the load-bearing factA 1M-token context sounds ruinous, but K3’s KDA:MLA hybrid is what makes serving viable: only 24 of 96 layers are MLA (the other 72 are Kimi Delta Attention — recurrent, near-zero fixed state). 24 × (512+64) × 2 B × 1,048,576 ≈ 28 GiB. An all-MLA model would be ~108 GiB/user (→ ~4× fewer users); a naive multi-head model would be hundreds of GiB/user (→ you could not seat one user at 1M/FP16). The architecture, not the hardware, is what makes concurrency affordable.

The count: users = (total VRAM × ~0.9 overhead − 1,450) ÷ 28. The ~0.9 reserves ~8–10% for activations, NVLink/NCCL comm buffers, KV paging, and fragmentation. Note this counts users each actively generating at a full 1M-token context, right now — the worst case. It excludes multi-turn KV parked in ordinary DDR between turns (a separate, much larger pool).

02Why 8× B200 is a non-starter — the model won’t even fit

Shipping HGX/DGX B200 carries 180 GB per GPU (NVIDIA’s DGX B200 spec: 1,440 GB ÷ 8; the 192 GB figure was the 2024 launch “up to” number — see flags). So an 8-GPU node is 8 × 180 = 1,440 GiB — less than the 1,450 GiB model itself. A single 8× B200 box cannot hold Kimi K3 at all, let alone one user’s KV. That is the precise reason the off-the-shelf answer starts at sixteen GPUs, and why K3 is forced to span more than one 8-GPU NVLink island.

04Why users scale faster than price — the super-linear win

From 16× B200 to NVL36 the price roughly doubles, but the user count roughly quadruples (42 → 165). That is not a rounding artefact — it is the fixed cost amortizing:

The 1.45 TiB model is a fixed tax paid once. On the small box it swallows half the VRAM; on the big box it is a rounding error — so the KV pool (and thus users) grows faster than the GPU count. Cost-per-user falls ₪80k → ₪39k → ₪28k (from ~₪63k if you self‑import the 16× box). This is exactly the observation that “serving more users on the same silicon is more economical” — quantified. The optimal machine for a serving business is the largest single memory pool you can afford, not the cheapest box that technically runs the model.

05The topology fork — why there is no clean “16× B200”

Your two goals — cheapest, and highest shared bandwidth between all GPUs — genuinely conflict, because NVIDIA offers NVLink domains in exactly two sizes:

  • NVLink domain sizes are 8 or 72 — nothing between. There is no “NVL16” for B200 (verified against NVIDIA’s NVLink spec table and the absence of any such SKU across NVIDIA, Supermicro, Dell, Gigabyte). To exceed 8 Blackwell GPUs in one NVLink domain you must step to the GB200 NVL36 / NVL72 rack (Grace + Blackwell superchips).
  • Cheapest 16× B200 = two 8-GPU islands + InfiniBand. Within each island every GPU pair runs at 1.8 TB/s; between islands it drops to one 400 Gb/s IB rail per GPU — an ~18× bandwidth cliff. Since K3 (1.45 TiB) cannot fit in one 1,440 GiB island, the model must straddle that seam, and every forward pass crosses it. The 42 users are VRAM-resident but throughput-throttled.
  • The literal 16-GPU single-domain box that exists is HGX B300 NVL16 — but that is Blackwell Ultra (B300, ~2.1 TB HBM3e across 8 packages), not sixteen physical B200s. It is the “16 in one NVLink fabric” product; it is simply a different (newer, denser) part.

So for genuinely sharing memory across all the GPUs — which is what sharding a 2.8 T model actually needs — the honest options are accept the IB seam (16× B200, cheap) or buy a single NVLink domain (NVL36+, pricier but cheaper per user).

06The Israel cost basis

  • FX: 3.05 ₪/USD — Bank of Israel official rate, cross-checked against ECB/others (all 3.04–3.05). The shekel is historically strong right now (vs ~3.6–3.7 in 2023–24), which flattens ILS prices ~15% below older intuition.
  • Import stack = price × 1.18, nothing else. VAT is 18% (raised from 17% on 2025-01-01). Servers (HS 8471) enter at 0% customs duty on two independent bases — the WTO Information Technology Agreement (Israel is a participant) and the US–Israel FTA. VAT is recoverable input tax for a registered buyer.
  • Anchor check: ₪1.6M incl. VAT for an 8× B200 + 2 TB server ÷ 1.18 = ₪1.356M ex-VAT ≈ $445k at 3.05 — essentially US street parity for a Supermicro-class 8× B200 build (~$449k). The anchor is credible and, at today’s strong shekel, tight (not inflated).
  • Global 8× B200 price band the anchor sits in: cheapest listed web config $338k (Exxact, 2 TB, no fabric NICs); realistic band $340–450k; DGX‑branded $515–580k; the bare HGX B200 8‑GPU baseboard alone ~$306–325k (≈ $38–40k/GPU). So the ₪1.6M local anchor (~$445k) runs ~24% above the cheapest US self‑import (~$338k → ~₪1.22M landed) — the price of buying local.

07The point, finished — and where STRATA comes in

All three machines above pay for their capacity in HBM3e — the most expensive memory per gigabyte in existence. Yet look at what the per-user cost actually buys: ~28 GiB of KV cache, which is just capacity that must be read at a bandwidth the GPU can hide latency behind. It does not need HBM’s bandwidth density; a GPU hides memory latency under thousands of threads regardless. That is precisely the gap the rest of this repository exploits:

Why cheap DDR5 is the endgame of this economicsThe KV cache and the weights both need capacity at adequate, latency-hideable bandwidth — exactly what commodity DDR5 delivers at ~5–10× lower cost per GB than HBM/GDDR7. A custom converter that fronts a DDR5 warehouse behind a GPU-facing memory interface (strata_v_design.html) serves the same K3 KV pool for a fraction of the per-GB cost — turning the ₪28–80k/user of the HBM machines into something far lower. And DDR5 gives this up nothing on latency: at the device level it is equal-or-lower latency than GDDR7 (see strata_v_design.html §12).

So the WhatsApp-thread claim — “NVIDIA could easily build a 2 TiB card to run K3, and it comes out cheap” — is directionally right about the physics (capacity is depth behind the pins; bandwidth is parallelism across many cheap chips; 1.79 TB/s is a low bandwidth). The word doing unfair work is easily: it requires a controller you own (custom silicon, ~$50M+ NRE) and a deterministic converter — you cannot retrofit a stock card (rtx5090_strata_v_retrofit_verdict.html). But the economic prize the thread is reaching for is real and large: the cost of serving a frontier MoE is dominated by memory capacity, and capacity is the one axis where HBM is worst and DDR5 is best. The off-the-shelf NVIDIA path prices that lesson at ₪28–80k per user; the custom-DDR5 path is the argument for driving it far lower.

08Honest flags

  • 180 vs 192 GB: shipping HGX/DGX B200 is 180 GB/GPU per NVIDIA’s current DGX B200 spec and Dell’s OEM listing; 192 GB was the 2024 launch “up to” figure (a documented NVIDIA inconsistency; GB200 rack parts are binned ~186 GB). At 192 GB the 16× count rises to ~48. Confirm the actual parts on any quote.
  • NVL36 / NVL72 pricing is analyst estimate (Morgan Stanley / TrendForce ~$1.8M / ~$3M per rack), not a vendor quote — no vendor lists a price; treat ±25%.
  • VRAM-resident ≠ throughput-servable. This is the capacity ceiling, not a tokens/s promise: 165–380 users each at a full 1M-token context is an enormous prefill/attention compute load. VRAM lets them reside; sustaining good tokens/s for all of them at once is a separate, compute- and bandwidth-bound question.
  • MTP (+18 GiB) assumes K3 ships an MTP module (no K2-line model did); its per-user cost is negligible (speculation-window only). Vision adds ~1 GiB resident, zero extra KV.
  • Prices are point-in-time, DRAM-crunch, FX 3.05 — volatile.

Bottom line. Serving Kimi K3 at 1M-token full quality is a VRAM-capacity game: ~1.45 TiB fixed + ~28 GiB per active user. A cheap 16× B200 (2 islands + IB) seats ~42 users for ~₪3.35M but throttles them across an InfiniBand seam; a single-NVLink-domain GB200 NVL36 seats ~165 for ~₪6.5M — and, because the model is a fixed cost amortized once, that is ~half the cost per user (~₪39k vs ~₪80k), with NVL72 lower still (~₪28k). The economic lesson — the cost of frontier inference is dominated by memory capacity — is the whole reason the custom-DDR5 converter in this repository exists.
