committed engineering design · codename STRATA‑V

A DDR5 memory‑virtualizer

A silicon converter hierarchy that makes ~600 ordinary DDR5 dies appear to a GPU as a single, impossible 2.25 TiB / 1.79 TB/s memory chip — that no real device could be.

Backing: 18 × MRDIMM‑12800 (128 GiB). Front: a 512‑pin, 28 Gb/s interface identical in pin count and bandwidth to an RTX 5090’s. Assumes custom 3‑nm converter silicon that can emulate any transfer at the memory‑controller clock. A hypothetical design record. Compiled 2026‑07‑18.

The realization behind this design: pins buy bandwidth, capacity is depth, and 1.79 TB/s is a low bandwidth. Once you accept those, a converter can aggregate a huge pool of cheap, deep DDR5 and present it to a GPU as one flat, fast, impossibly‑large device — because no single DDR5 chip provides this bandwidth, but ~18 channels interleaved do.

What it is

One virtual chip. ~600 real dies. The converter is the illusion.

To the GPU, STRATA‑V is a single memory device with a flat 41‑bit address space, 2.25 TiB, 1.79 TB/s, and GDDR‑class fixed‑beat timing. Inside, a converter hierarchy stripes that address space across 18 DDR5 channels (~576 × 32 Gb dies), interleaving so a sequential read fans across all channels at once. It does not make DDR5 faster; it parallelizes it — 18× a channel, 287× a single die — and hides DDR5’s variability so the front can run a deterministic beat.

Tier V — what the GPU sees

Tier 2 — DRAM

The root converter presents the impossible‑chip face and stripes the flat address across channels (a RAID‑0 stripe over DRAM). The channel converters each schedule ~3 DDR5 channels and, crucially, hide refresh and bank conflicts behind a small reorder buffer so the front sees deterministic timing. The whole thing can collapse onto one large converter die if pin/thermal budgets allow (fewer hops, lower latency), or split into the root+leaf hierarchy shown (+~5 ns/level).

04Where the bandwidth comes from — honestly

The converter does not overclock DDR5. It interleaves:

  • 1 DDR5 die = 6.4 GB/s; 1 MRDIMM channel = 102 GB/s; 18 channels striped = 1.84 TB/s.
  • The root converter maps consecutive cache lines onto different channels, so a sequential read fans across all 18 at once — the sum of their bandwidth.
  • Ceiling = the sum of the backing, minus overhead. A buffer can burst briefly above that; sustained bandwidth is exactly the aggregate. The converter aggregates bandwidth — it can never manufacture it. Front 1.79 TB/s < backing 1.84 ⇒ the front is the steady throttle.

So “higher bandwidth than DDR5 provides” is true per component (18–287×) and comes entirely from quantity + interleave — not from any single chip doing more than it can.

The 1.79 TB/s target is arbitrary — chosen only to match the RTX 5090. Bandwidth and capacity are independent knobs: set any front bandwidth by the channel count (N × per‑channel), and any capacity by the depth (channels × 3DS stack). 1.79 TB/s and 2.25 TiB are simply the point we picked.

06The catches — the honest accounting

  1. Determinism / refresh is THE engineering problem — not latency, not bandwidth. A fixed‑beat front has no “please wait,” so every channel’s tRFC refresh (~hundreds of ns) and bank conflicts must be scheduled or buffered out of the demand path by the channel converters (stagger refresh across channels; keep a few‑µs reorder buffer). Solvable, but it is the part you actually have to engineer — and get wrong and you silently latch stale data.
  2. It is an appliance, not a card. 18 tall MRDIMMs + converters + compute module ≈ a ~40 × 40 cm baseboard, ~360 W of DRAM. That is the price of 2.25 TiB.
  3. Front pins = a 5090; total system pins ≠ a 5090. The compute die keeps 512 pins; the ~2,000 backing pins live on the converters. The pin count did not vanish — it moved to where it is short and cheap.
  4. MRDIMM‑12800 is DDR5 Gen2 (sampling as of 2026); the ships‑today build uses DDR5‑6400 RDIMM across 35 channels instead of 18.

07Open questions & unverified items

  • Absolute RCD / data‑buffer / reorder‑buffer latency adders (only “RCD ~1 nCK” is firm; the rest are estimates).
  • The GPU‑visible ~200–300 ns figure is an estimate for this configuration, not a measurement.
  • Exact 128/256 GiB MRDIMM part numbers and the tall‑module height/thermal fit.
  • The refresh‑scheduling worst case (all‑channel refresh collision under a full‑bandwidth stream) needs a worked timing budget before commit.

Part II — protocol & correctness, from first principles

Sections 01–07 fixed what STRATA‑V is. This part asks whether it can correctly and robustly present itself as the memory device it claims to be — derived from scratch, assuming nothing obvious, and grounded in the specific (and partly arbitrary) choices already made: a 512‑pin, 28 Gb/s front matching an RTX 5090; ~18–24 MRDIMM‑class DDR5 channels behind the converter hierarchy; 2.25 TiB at 1.79 TB/s. Those two numbers are arbitrary anchors — 1.79 TB/s only to match a 5090, 2.25 TiB only to clear 2 TiB — and the reasoning holds for any target picked the same way.

08The interface, from first principles: two buses, one contract

Assume nothing. A DRAM interface is not one bus — it is two, on wildly different clocks, plus housekeeping. On the 5090‑class front we chose, the pins fall into:

BusPins (per device)RateCarries
Data (fast, hard)DQ[31:0] + WCK (write clock) + RCK (read clock)28 Gb/s PAM3the 1.79 TB/s payload — and the only part with tight signal integrity
Command / address (slow, easy)CA[n:0] + CK (command clock)~1–2 GHzthe commands (ACTIVATE, READ, WRITE, PRECHARGE, REFRESH, mode‑register set) and the bank/row/column address, multiplexed over cycles
HousekeepingCKE, RESET, ZQ, EDCslowenable, reset, impedance calibration, error feedback

One rule governs all of it, and it follows from what DRAM is: a DRAM array has a fixed physical access time and cannot negotiate, so the interface has no flow control. The controller issues READ and samples the data an exact, pre‑agreed CL cycles later — no “data ready” wire, no way to stall. This single fact is the whole problem: everything the converter must do reduces to honoring one fixed‑latency contract, and every risk reduces to missing a beat under it.

09Extending the address space to our (arbitrary) 2.25 TiB

Capacity is not carried by the data pins — it is carried by address bits on the slow CA bus. A 512‑bit word bus of capacity C needs log₂(C / 64 bytes) word‑address bits:

  • Our 5090 anchor, 32 GiB → ~29 word‑address bits.
  • Our chosen 2.25 TiB → ~35 bits — exactly +6 bits (2⁶ = 64×, matching the arbitrary 64× capacity jump).

Those 6 bits ride the CA bus (more row/bank address, over a few extra CA pins or cycles). Because CA is the slow bus, adding pins there is electrically free — nothing like routing another 28 Gb/s PAM3 lane. The fast data pins stay byte‑for‑byte a 5090. The whole capacity extension lands in the easy domain. It makes the front a GDDR7‑derived protocol rather than stock GDDR7 — free here, because we own both ends of the link.

10Can it always serve on time? The core proof

The reframe that settles it: the converter does not race the interface’s native deadline hoping to win — it authors the deadline.

From first principles, CL, tRCD, tRC, and the refresh interval are not constants of physics — they are parameters we program into the controller we designed. So we set them to values our specific backing (~18–24 MRDIMM channels) can always satisfy, and the controller can then only ever issue command streams legal under that contract. On‑time service stops being a race and becomes a property of construction. The ~10–30% latency premium (§05) is exactly the size of the margin we buy to make this true.

Proof against the worst‑case legal stream

The nastiest thing the controller can legally emit: peak READs (tCCD‑apart, bank‑interleaved), a row‑miss on every access (max activate rate), refresh interspersed. Each demand is already answered by a choice tied to our architecture:

Worst‑case demandWhy it is already covered
Sustained read rate = 1.79 TB/sbacking over‑provisioned (this is why ~20–24 channels, not 18) → sustains 1.79 TB/s net of refresh
Max activate rate (tRRD/tFAW)DDR5 aggregate has hundreds of banks; set virtual tRRD/tFAW ≥ DDR5’s
Same‑bank cycling (tRC)set virtual tRC ≥ DDR5 tRC → the row is always ready by the beat
Bank aliasingbijective address map: distinct virtual banks → distinct physical banks — aliasing impossible by design
A read landing during refreshcannot happen: the controller respects its own refresh schedule (which we set) and does not read during the window; the converter refreshes DDR5 then
Prefetch at full tiltjust more reads at the same CL — no new deadline; if anything it helps (predictable)
The two — and only two — ways it can failBoth are design‑time bounded checks, verified on paper before tape‑out, never runtime surprises:
  1. The required CL must fit the controller’s CL field. Our serviceable CL (≈ backing access + converter hop + margin, ~50–100 ns) must be encodable — within range for our short backing path, and since we design the controller we can widen the field.
  2. The DDR5 refresh demand must fit the schedulable refresh budget. The tight one: our 256 GiB 3DS parts have a large tRFC, so a higher refresh tax than a 24 Gb GDDR7 die. Mitigate with per‑bank refresh (one DDR5 bank busy at a time), lower‑density parts, more channels, or a wider custom refresh range.
Pass those two and no runtime command stream — peak streaming, worst‑case row‑miss flood, aggressive prefetch, refresh — can produce a late serve.

11Functional equivalence — and the one inherent gap

The general question, not tied to our MoE target: is there anything the memory we replace (a 5090’s 32 GiB GDDR7) can do that STRATA‑V cannot, for any use case?

CapabilityReproduced?
Every operation: R / W / ACT / PRE / REF / MRS / trainingYes — plain DRAM ops; atomics live in the GPU’s L2, not the array
Sustained bandwidth, any access patternYes — over‑provision the backing
Bank parallelism, small‑access granularity, writesYes — bijective map / single‑channel routing / write buffer
Deterministic fixed‑beat timingYes — by the §10 construction

Functionally, STRATA‑V is a strict superset: everything the 32 GiB does, plus our 64× capacity. There is exactly one inherent difference — a latency premium (~10–30%), from the translation hop and physically‑further, denser DDR5. By Little’s law (throughput = concurrency ÷ latency) it reduces effective bandwidth only for latency‑bound, low‑parallelism access — a CPU‑style pattern (pointer‑chasing a dependency chain). GPUs, and our MoE inference target, run high memory‑level parallelism and hide it completely.

The honest bottom line of the equivalenceFor every workload a GPU actually runs, STRATA‑V is ≥ the memory it replaces on every axis — same bandwidth, same protocol, 64× the capacity. The only thing real GDDR7 wins is the last sliver of latency in a regime GPUs are built to avoid — and that premium is the price of the trade (bandwidth from many cheap DDR5 chips instead of graphics DRAM), not a capability surrendered.

12The substrate — DDR5 vs GDDR7 characteristics

STRATA‑V’s whole premise is swapping GDDR7 for a DDR5 warehouse — honest only if DDR5 does not quietly cost latency. It does not: at the device level DDR5 is equal‑or‑lower latency than GDDR7, and better on every axis except the one GDDR exists for. First disambiguate the word, because it is quoted two incompatible ways.

  • Device‑level latency (the DRAM chip): both use the same 1T1C array, so core row mechanics (tRCD, tRAS, tRP) are near‑identical in nanoseconds. For DDR5: row‑hit (page open) ~16 ns (tCL); row‑empty ~32 ns (tRCD+tCL); row‑miss ~49 ns (tRP+tRCD+tCL). tCL has stayed ~16 ns for four generations — as MT/s rises, CL‑in‑cycles rises to match. GDDR7’s exact timings live in the gated JESD239, but directionally its first‑word latency sits modestly above DDR5’s: same array, plus a bigger PHY tax (28 Gb/s PAM3 needs more serialisation, equalisation, WCK/RCK training, and a deeper prefetch/CRC path than DDR5’s slower NRZ link).
  • System‑visible latency (what benchmarks quote): ~335 ns on a GPU vs ~80–100 ns for CPU DDR5. That ~3–4× gap is almost entirely the compute‑side hierarchy — L2‑miss + a huge crossbar + throughput‑tuned MC queuing — not the DRAM chip. Swap the memory type and most of that 335 ns would not move.

Why this is load‑bearingDDR5’s equal‑or‑lower array latency is what lets the converter’s ~15–25 ns translation hop cancel out (§05): net GPU‑visible latency stays comparable to a real GDDR7 card. You are not trading latency for capacity — you are trading GDDR7’s one genuine advantage, bandwidth per pin, which the converter rebuilds by interleaving ~45 cheap channels. Everywhere else — latency, capacity, power, cost, reach — DDR5 was already ahead. The “GPU memory is slow” reputation is the GPU’s hierarchy talking, not the DRAM.

Bottom line. STRATA‑V is a silicon interface that pretends to be a single 2.25 TiB / 1.79 TB/s DRAM chip that cannot physically exist — synthesized from ~576 ordinary DDR5 dies by an interleaving converter hierarchy. To a GPU it is indistinguishable from a colossal GDDR device: same 512 pins, same bandwidth, same fixed‑beat protocol, prefetch and bursts intact, latency comparable. The bandwidth is real aggregation, the capacity is real depth, and the only genuinely hard part is the converter’s refresh/conflict scheduling — the discipline that keeps the illusion deterministic.
